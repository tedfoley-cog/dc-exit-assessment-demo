# Implementation Plan — Migration-Readiness Assessment Demo

## 1. What the demo proves

A data-center exit at portfolio scale is bottlenecked on per-application
engineering knowledge: what each app does, what couples it to the data
center, and what it needs to reach a cloud landing zone. This demo shows
parallel Devin sessions producing, in under an hour and with read-only
repo access, the migration-readiness assessment artifact that a large SI
would take months (and a large budget) to produce — and the customer owns
it outright. The per-app remediation backlogs the sessions generate are the
migration factory's work queue, which is the natural paid next step.

## 2. What Devin does live

One parallel Devin session per legacy repo runs the readiness-assessment
playbook (index into DeepWiki, apply cloud-readiness rules, produce a
readout + remediation backlog + disposition), then a coordinator session
aggregates the readouts into a portfolio disposition table.

## 3. Stack and rationale

| Component | Choice | Source confirmed against |
|---|---|---|
| Java tranche app | Struts 1.2.9 on Java 7, Maven WAR | Apache Struts 1.x user guide & DTDs (struts.apache.org/struts1); Struts 1 EOL announcement (CVE-class: unfixable) |
| ColdFusion tranche app | Adobe CF9-era CFML (`Application.cfc`, `cfquery`, `cffile`, `cfftp`, `cfmail`) | Adobe ColdFusion 9 CFML Reference (helpx.adobe.com) |
| COBOL tranche app | Enterprise COBOL 85-style batch + JCL, VSAM KSDS, GDG, IDCAMS | IBM Enterprise COBOL for z/OS Language Reference; IBM JCL Reference; validated with GnuCOBOL `cobc -std=ibm -fsyntax-only` |
| Readiness rules | YAML ruleset modeled on Konveyor analyzer rules (categories: EOL runtime, vulnerable deps, hardcoded endpoints, local FS coupling, plaintext credentials, missing tests) | github.com/konveyor/rulesets |
| Dispositions | Gartner/AWS "R" taxonomy: rehost / replatform / refactor / retire / replace | AWS migration strategy docs ("7 Rs") |
| Aggregation | Small stdlib-only Python script that renders the portfolio disposition table + static HTML readout from per-app readout front-matter | — |

## 6. Runtime plan

The demo runtime is Devin itself: the presenter kicks off 3 parallel
sessions (one per tranche repo) with the playbook prompt, plus this hub
repo. Each session needs only read access to its repo. The tranche repos
are genuinely runnable/checkable (Struts app compiles under JDK 8; COBOL
syntax-checks under GnuCOBOL; CFML passes tag-balance checks) so findings
are grounded in real code, but no app runtime is needed for the demo.
`scripts/aggregate_readouts.py` runs in the coordinator session and renders
the portfolio table (verified locally against fixture readouts).

## 7. Visual artifact plan

Primary visuals are the use case's native outputs (tier 2 of the
hierarchy): the per-app readiness readouts with file:line evidence, the
portfolio disposition table, and the DeepWiki pages that materialize per
repo. A static self-contained HTML portfolio report is generated by the
aggregation script. No live dashboard: the demo is a one-shot assessment
whose result is a set of documents, so it fails the Dashboard Decision
Gate's "tracking is the point" criterion.

## 8. CI plan

- Hub repo: ruff lint of the aggregation script + YAML validation of the
  ruleset and portfolio file + a fixture-driven run of the aggregator.
- legacy-supplier-portal: `mvn package` on Temurin 8.
- legacy-workorder-cf: CFML tag-balance sanity check.
- legacy-inventory-cobol: GnuCOBOL `cobc -fsyntax-only -std=ibm`.

## 9. Risks and unknowns

- The live demo depends on org concurrency limits allowing 3–4 parallel
  sessions; confirm before the call.
- DeepWiki indexing time for fresh repos varies; kick indexing off at the
  start of the demo (or pre-index the hub repo only, keeping tranche repos
  cold for authenticity — presenter's choice).
- Boeing-specific MRC standards are not public; the ruleset uses
  Konveyor-style generic cloud-readiness rules and marks the MRC hook
  points where customer standards would slot in.
